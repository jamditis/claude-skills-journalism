Claude Code Plugin

AutoContext

A Claude Code plugin that gives your AI assistant a memory for each project.

Every time you correct Claude — "no, use the other API", "you forgot to clear the cache" — autocontext saves that as a lesson. Next session, it loads those lessons before Claude starts working, so the same mistakes don't happen twice.

The lesson loop above captures project-specific knowledge. Skill evolution takes it further: lessons that apply to a skill itself get folded back into the skill file, so the skill permanently improves.

How confidence works

Every lesson has a confidence score from 0.0 to 1.0. The score determines whether a lesson gets loaded, how prominently it appears, and when it fades away.

New lesson: 0.5

Auto-extracted lessons start at 0.5 confidence. Lessons seeded from CLAUDE.md start at 0.6 (slightly higher since they were already documented). This puts new lessons in the "visible but not trusted" range.

Validation: +0.1

Each session that loads a lesson without contradicting it bumps confidence by 0.1 (capped at 1.0). A lesson validated across 5 sessions reaches 1.0 — the system's highest trust level.

Contradiction: −0.2

If a session contradicts a lesson (the user corrects the behavior the lesson describes), confidence drops by 0.2. Contradictions carry twice the weight of validations — one correction matters more than two confirmations.

Staleness decay: −0.1 per 30 days

Lessons not validated within the staleness window (default: 60 days) begin decaying. A lesson ignored for 3 months loses 0.1 confidence. This prevents old, possibly-outdated knowledge from cluttering the session.

Confidence threshold

Lessons below the threshold (default 0.3) aren't loaded at session start. They still exist in lessons.json and can be reviewed with /autocontext-review, but they don't consume session context.

High confidence0.7 – 1.0

Loaded first, shown prominently in playbook

Medium confidence0.3 – 0.7

Loaded if space permits, ranked below high-confidence

Below threshold0.0 – 0.3

Not loaded — available only via /autocontext-review

Why time-based decay?

Session-count-based decay would need a global counter synced across all machines and developers. Time-based decay uses only the last_validated timestamp on each lesson, which merges cleanly across branches. No coordination needed.

How lessons are captured

The UserPromptSubmit hook scans every message for correction patterns. No LLM call — just fast pattern matching on the text you type.

Correction patterns

When you correct Claude mid-session, the hook captures your message as a lesson candidate. These phrases trigger detection:

"no, use X instead"

Direct correction — Claude did the wrong thing, and you're telling it the right way

"that's wrong" / "don't do that"

Negative feedback — Claude made an error worth remembering

"actually it should be..."

Correction with the right answer inline

"remember that..." / "keep in mind..."

Explicit instruction to retain knowledge

"the correct way is..." / "you forgot that..."

Teaching moment — Claude missed something it should know

Three-phase extraction

Lesson extraction is split across three lifecycle events to avoid the SessionEnd timeout constraint (default 1.5 seconds):

Phase 1 — During session

Detect and queue

UserPromptSubmit pattern-matches your corrections and appends raw candidates to pending-lessons.json. No LLM call — this takes milliseconds.

Phase 2 — Session end

Save candidates

SessionEnd writes the candidates file and bumps validated_count on loaded lessons. Pure file I/O — fits in the 1.5s timeout.

Phase 3 — Next session start

Curate with LLM

SessionStart calls claude -p with the curator prompt. It decides which candidates are project-specific, actionable, and worth keeping. Approved lessons get added to lessons.json at confidence 0.5.

What the curator rejects

  • General programming knowledge ("use git for version control")
  • Vague advice ("be careful with X")
  • One-time tasks that won't recur
  • Anything containing secrets, API keys, or PII

Cross-developer merging

When two developers both accumulate lessons independently, git pull triggers a custom merge driver that resolves conflicts field by field — no manual conflict resolution needed.

How the merge driver works

The driver (scripts/merge-driver.py) gets three versions of lessons.json: the common ancestor, yours, and theirs. It merges them with union semantics — both sides' new lessons are kept, and conflicts on shared lessons are resolved per field.

FieldResolutionRationale
validated_countAdditiveBoth devs may have validated independently
confidenceHigher value winsMore evidence = more trust
last_validatedMost recentLatest validation is most relevant
deletedtrue winsDeletion is intentional
tagsUnionTags are additive, never lost
text, contextFlag needs_reviewContent changes need human review
supersedesNon-null winsIf both non-null, flags for review

Tombstone deletion

Deleted lessons aren't removed from the array — they stay with "deleted": true. This prevents a deleted lesson from being re-added by another developer's stale copy. Tombstones accumulate until you clean them up with /autocontext-review.

Safety net

If lessons.json fails to parse after a merge (bad JSON from a manual edit or broken merge), the SessionStart hook warns and skips lesson injection rather than crashing the session. The driver must be installed locally — .gitattributes only declares which files use it. If missing, SessionStart warns you to run /autocontext-init.

Identity attribution

Each lesson records its created_by identity. Identity comes from config.local.json (gitignored), the global ~/.claude/autocontext.json, or falls back to git config user.email + hostname. Never stored in the shared config.

Lesson anatomy

Each lesson is a JSON object with fields for content, scoring, attribution, and lifecycle management.

{
  "id": "lesson_a1b2c3d4",
  "schema_version": 1,
  "category": "efficiency",
  "text": "POST /api/ideas/ requires trailing slash",
  "context": "Dashboard Board API, FastAPI routes",
  "confidence": 0.9,
  "validated_count": 3,
  "last_validated": "2026-03-13T22:00:00Z",
  "created": "2026-03-10T14:00:00Z",
  "created_by": "joe@officejawn",
  "supersedes": null,
  "deleted": false,
  "tags": ["api", "fastapi", "machine:officejawn"]
}
id

Unique identifier. Format: lesson_ + 8 hex chars. Used by the merge driver to match lessons across branches.

category

One of efficiency (API patterns, conventions), codebase (schema, architecture), or optimization (performance, benchmarks). Determines playbook grouping.

supersedes

Points to the ID of a lesson this one replaces. The old lesson gets deleted: true. This creates an audit trail — you can trace how knowledge evolved.

tags

Free-form strings for file paths, module names, and concepts. Tags prefixed with machine: are filtered on non-matching machines — machine:officejawn only loads on officejawn.

schema_version

Currently 1. Enables future format migrations without breaking existing lesson files.

Test quality rules

Five built-in rules catch common test anti-patterns. Two are deterministic (regex-based, no LLM cost). Three are advisory (LLM-based, run as part of the PreToolUse hook).

deterministic

No bare assertions

Regex scan for weak assertions: assert True, assert result is not None, assert len(result) > 0, self.assertTrue(True).

These assertions prove the code ran without errors, but don't verify it produced the right output.

deterministic

No happy path only

Counts test method names. If all start with test_success / test_valid / test_happy and none contain error / fail / invalid / edge, it flags the file.

Good test suites cover failure modes, not just the golden path. May produce false positives on small test files.

advisory

Tautological test check

LLM checks whether assertions are derived from reading the implementation rather than from the task or spec.

A test that copies the implementation's output as its expected value will always pass — even if the implementation is wrong.

advisory

No mock everything

LLM checks if mocked return values are the same values being asserted. If so, the test is verifying the mock, not the code.

When mock.return_value = X and the assertion is assert result == X, nothing is being tested.

advisory

Test independence

LLM checks whether the test would still pass if the feature under test didn't exist.

A test that passes without its feature code is testing infrastructure, not behavior.

Persistence modes

  • auto_curated — Lessons go through the curator LLM before being saved. Recommended for most projects.
  • ask_before_persist — Curated lessons are shown via /autocontext-review for manual approval.
  • auto_persist_all — Every correction is saved directly. No curator, no review. Use for fast-moving solo projects.

Use cases

The plugin is project-agnostic. Any codebase where Claude Code sessions repeat the same mistakes benefits.

Content and translation pipelines

Lessons accumulate about which content types get mis-scored, terminology patterns, and per-language quirks. Reduces human reviewer corrections over time.

Multi-developer game projects

Schema shapes, authority models, build sequences, and handoff protocols sync via git so both developers' Claude instances share the same understanding of the codebase.

Multi-service infrastructure

API patterns (auth headers, endpoint quirks), status sync rules, and processing workflows. Prevents repeated mistakes on service integration points.

Any large codebase

Performance characteristics, architectural decisions, test patterns, and deployment gotchas. The optimization category tracks build/test timing baselines and flags regressions.

FAQ

Does this require API keys or external services?

No. Everything runs locally via Claude Code hooks. Curation uses claude -p (a subprocess call to your existing Claude Code installation), not a direct API call. No tokens, no accounts, no daemons.

What's the token cost?

SessionStart and SessionEnd are command hooks — no LLM cost. UserPromptSubmit is pattern matching — no LLM cost. The PreToolUse hook is prompt-based and has token cost proportional to the number of Edit/Write/Bash calls. If cost is a concern, disable it with pretooluse_hook: false in config.

Can I use this with an existing CLAUDE.md?

Yes. /autocontext-init offers to seed lessons from your CLAUDE.md. It extracts project-specific gotchas (skipping general programming knowledge) and creates lessons at confidence 0.6. You can review each one before saving.

What happens if lessons.json gets corrupted?

The SessionStart hook validates JSON before loading. If parsing fails (from a bad merge, manual edit, or corruption), it warns and skips lesson injection. The session continues normally — you just don't get lessons that time.

How is this different from just putting everything in CLAUDE.md?

CLAUDE.md is static — you write it, it stays forever. Autocontext is dynamic: lessons are captured from corrections automatically, scored by confidence, decayed when stale, and merged across developers without conflicts. A CLAUDE.md entry has no way to know it's outdated.

What are the non-goals?

No semantic search (tag-based matching is simpler). No training or fine-tuning (knowledge is prompt-level, not weight-level). No MCP server (hooks are lighter and always available). No external services or daemons.

Disabled by default. Enable via /autocontext-setup step 11, or set skill_learning.enabled: true in config. Zero overhead if you don't use it.
